Step 1: Open server.js

Your current file looks like this:

const express = require("express");
const cors = require("cors");

const roomRoutes = require("./routes/roomRoutes");
const codeRoutes = require("./routes/codeRoutes");

const app = express();

app.use(cors());
app.use(express.json());

app.use(roomRoutes);
app.use(codeRoutes);

app.get("/", (req, res) => {
    res.send("Backend Running");
});

app.listen(5000, () => {
    console.log("Server running on port 5000");
});
Step 2: First Modification
Add two new imports
const http = require("http");
const { Server } = require("socket.io");

Now your imports become

const express = require("express");
const cors = require("cors");
const http = require("http");
const { Server } = require("socket.io");

const roomRoutes = require("./routes/roomRoutes");
const codeRoutes = require("./routes/codeRoutes");
Why?

Let's understand.

This line
const http = require("http");

means

Give me Node's HTTP Server.

Think of this as the actual building.

Earlier Express was saying

Don't worry, I'll create the building for you.

Now we're saying

No.

We want to own the building ourselves.

Why?

Because Socket.IO wants to live inside that building.

This line
const { Server } = require("socket.io");

imports the Socket.IO Server class.

Think of it as hiring a communication manager.

The HTTP server owns the building.

Socket.IO manages real-time conversations inside that building.

Step 3

After

const app = express();

add

const server = http.createServer(app);

Now your code becomes

const app = express();

const server = http.createServer(app);
What happened here?

Earlier

Browser

↓

Express

Now

Browser

↓

HTTP Server

↓

Express

This is MUCH more professional.

Step 4

Delete

app.listen(5000, () => {
    console.log("Server running on port 5000");
});

Replace it with

server.listen(5000, () => {
    console.log("Server running on port 5000");
});
Did We Add Socket.IO?

No.

And this is intentional.

Many tutorials would already write

const io = new Server(server);

We're not doing that yet.

Why?

Because I want you to verify that after replacing app.listen() with server.listen(), nothing breaks.

A senior developer makes one logical change at a time, runs the project, and confirms it still works before adding the next feature.

What Should Happen?

Run the backend:

npm start

or

npm run dev

Then visit:

Expected response:

Backend Running

If everything works, then congratulations—you've successfully replaced Express's hidden HTTP server with your own HTTP server. Functionally nothing changed, but architecturally you've prepared the project for Socket.IO.

🧠 Why didn't we add Socket.IO yet?

Because software engineering is about building confidence in small, verified steps.

Today you learned something professional:

Never introduce two major changes at once.

If the application breaks after changing five things, it's hard to know why.

If it breaks after changing one thing, the cause is obvious.

This debugging mindset will save you countless hours on real projects.
